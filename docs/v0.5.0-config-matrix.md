> **Purpose (R9 mitigation):** prevent doc-drift recurrence by maintaining a
> single source of truth for every new PluginConfig field and every new
> invariant. HF1-HF3 hotfixes were caused by exactly this drift class in
> v0.4.0.
>
> If you change a default, validation, or wording for any of these,
> update THIS doc + the related code + the related test in lock-step.

## Invariants (3 new)

| Invariant | Statement | Enforcement | Where defined | Tests |
|-----------|-----------|-------------|---------------|-------|
| **INV-32** | Heartbeat thread NEVER blocks/kills auto run; first-failure stderr breadcrumb; counter persisted incrementally via queue → main thread → auto-run.json | `scripts/heartbeat.py:HeartbeatEmitter._emit_tick` + `auto_cmd._drain_heartbeat_queue_and_persist` | spec sec.2.7 | `tests/test_heartbeat.py::test_heartbeat_first_failure_*`, `tests/test_auto_progress.py::test_update_progress_drains_*` |
| **INV-33** | Per-stream timeout is last-resort kill (heartbeat 1st-line, watch 2nd-line, timeout 3rd-line, operator 4th) | `subprocess_utils.run_streamed_with_timeout` | spec sec.2.7 | `tests/test_subprocess_utils.py::test_t1_*` |
| **INV-34** | Timeout-vs-interval relationship: `timeout >= 5*interval` (clause 1); `interval <= 60` (clause 2); `interval >= 5` (clause 3); `timeout >= 600` (clause 4) | `config.py:load_plugin_local` (4 clauses with distinct error messages, ratio-first ordering) | spec sec.2.7 | `tests/test_config.py::test_inv34_clause_*` |

### W1 — INV-34 clause 1 defense-in-depth

In the default range `interval ∈ [5, 60]` and `timeout ∈ [600, ∞)`, INV-34
clauses 2 + 4 already imply clause 1: `5 * interval ≤ 5 * 60 = 300 ≤ 600 ≤
timeout`. Clause 1 is therefore mathematically subsumed and never fires
under default config. We **intentionally preserve it** as defense-in-depth
against future weakening of clauses 2 or 4 that could make clause 1 the
sole barrier preventing pathological ratios. Removing clause 1 would
require simultaneous re-evaluation of clauses 2 and 4 — a single-clause
audit must remain sufficient.

### W3 — Origin disambiguation jitter tolerance

Production default `DEFAULT_ORIGIN_WINDOW_SECONDS = 0.100` (100ms),
raised from the iter-1 baseline of 50ms in response to Checkpoint 2 iter
4 melchior + balthasar + caspar findings about Windows CI scheduling
jitter on loaded systems. Tests pass smaller windows (5ms) via the
`origin_window_seconds` keyword argument when the test wants
deterministic boundary behaviour, or larger windows (500ms) when they
need both writes to land inside the window despite jitter.

### W7 — Threading correctness as accepted-risk

Heartbeat (`scripts/heartbeat.py`) + Windows reader fallback (C2 in
`subprocess_utils.run_streamed_with_timeout`) introduce daemon threads
into otherwise single-threaded `auto_cmd` invocations. Threading
correctness is high-blast-radius but mitigated by:

1. Lock-protected singleton (`scripts/heartbeat.py`); no shared mutable
   state outside the lock.
2. Queue-based reporting (INV-32); only the main thread writes
   `auto-run.json`.
3. Event-interruptible tick loop (H2); the daemon exits cleanly when
   the context manager exits.
4. Single-thread `auto_cmd` invariant: the orchestrator never spawns
   peer threads beyond the heartbeat daemon and the streaming-pump
   reader threads.

Per Checkpoint 2 iter 4 balthasar, this is treated as **accepted-risk**:
no additional code change beyond the existing mitigations; the risk is
documented here so future maintainers can re-evaluate when the
single-thread invariant changes.

## Worked examples

### Example 1: Default config

```yaml
# (no overrides; all defaults applied)
```

Resulting validated config: `timeout=900s`, `interval=15s`,
`watch_interval=1.0s`, `origin=True`, `allowlist=("magi-*",)`. All INV-34
clauses satisfied (`900 >= 75` clause 1; `15 ∈ [5, 60]` clauses 2 + 3;
`900 >= 600` clause 4).

### Example 2: Aggressive CI config

```yaml
auto_per_stream_timeout_seconds: 600
auto_heartbeat_interval_seconds: 10
auto_origin_disambiguation: false
```

Validates: `600 >= 50` (clause 1), `10 ∈ [5, 60]` (clauses 2 + 3),
`600 >= 600` (clause 4 boundary). Origin disambiguation off for
parser-strict consumers.

### Example 3: INV-34 clause 1 violation

```yaml
auto_per_stream_timeout_seconds: 50
auto_heartbeat_interval_seconds: 15
```

Raises:
`ValidationError: INV-34 clause 1: auto_per_stream_timeout_seconds (50) must be >= 5 * auto_heartbeat_interval_seconds (15) = 75; got 50`.

### Example 4: INV-34 clause 2 violation

```yaml
auto_heartbeat_interval_seconds: 120
```

Raises:
`ValidationError: INV-34 clause 2: auto_heartbeat_interval_seconds must be <= 60s ...; got 120`.

### Example 5: INV-34 clause 3 violation

```yaml
auto_heartbeat_interval_seconds: 2
```

Raises:
`ValidationError: INV-34 clause 3: auto_heartbeat_interval_seconds must be >= 5s ...; got 2`.

### Example 6: INV-34 clause 4 violation (ratio satisfied but absolute floor failed)

```yaml
auto_per_stream_timeout_seconds: 75
auto_heartbeat_interval_seconds: 15
```

Satisfies clause 1 (`75 == 5 * 15`) but violates clause 4. Raises:
`ValidationError: INV-34 clause 4: auto_per_stream_timeout_seconds must be >= 600s ...; got 75`.

### Example 7: Allowlist bare wildcard rejected

```yaml
auto_no_timeout_dispatch_labels: ["*"]
```

Raises:
`ValidationError: auto_no_timeout_dispatch_labels: bare '*' rejected (would defeat timeout); use specific glob like 'magi-*'`.
